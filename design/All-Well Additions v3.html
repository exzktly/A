Five additions to the approved mockup

01

Plot theme — dark in-app, light on export

Two concerns, one mechanism. The plot canvas always renders in the dark theme that integrates with the app chrome. A persistent toggle in the figure header lets the user preview the publication theme in place; the export dialog then defaults to that theme but lets the user override per export.

How it behaves

  • Screen is the default and the canvas state. The toggle never persists between sessions — opening a file always lands on Screen.
  • Publication is a preview: the canvas re-renders white, fonts switch to publication-safe weights, axes thicken to 1.2 px, the trace palette shifts to higher-contrast publication variants (deeper blue, deeper red).
  • While Publication is active, the figure header shows a "preview only" warning chip so the user is never confused about why everything is white.
  • The export dialog reads the current preview state — if Publication is on, it pre-selects "Publication" so a one-click export gives what you see.
  • Transparent exists only in the export dialog (not as a preview) because there's no useful canvas state for it.
Implementation note. Matplotlib's rcParams is keyed to the theme; toggling the segmented control swaps rcParams and calls canvas.draw_idle(). The export reuses the same rcParams stack, so export and preview are guaranteed to match.

Statistics: SEM/SD & FOV/Spread

These lived in the old matplotlib toolbar and have no obvious new home. They're display options for aggregated data, not navigation. They belong with other display controls in the Properties panel — and get a quick-access popover in the figure header for users who reach for them constantly.

Two surfaces, one source of truth

  • Primary home: a new "Statistics" section in the Properties panel, between "Data" and "Appearance". Closed by default once configured.
  • Live preview value in the collapsed header ("SEM · spread") tells you the current state without expanding.
  • Quick popover: a "Stats · SEM" chip in the figure header opens the same three controls in a compact popover. Use when you toggle constantly during exploration.
  • Across · Replicates / FOV: renamed from "FOV/Spread" — the old term conflated the data axis (FOV vs replicates) with the display style (spread vs bars). Now the data axis is the Across control and the display style is Show.
  • Keyboard: E cycles error-bar types, ⇧E toggles Replicates ↔ FOV. Discoverable via tooltips on the segmented buttons.
Why move from the toolbar? The toolbar is for navigation (home/pan/zoom/save) — actions that take the figure somewhere. Stats toggles change what the data says. Mixing them confused users who treated SEM/SD like a pan tool ("how do I turn it off?"). Separating navigation from display fixes that.

Saved selections — unified panel

Replaces the legacy "replicate sets" and "bar groups" panels with one ordered list. Every saved selection has: a visibility toggle, color, name, well count, sub-item chips, and a kebab menu. Drag to reorder; rename inline; right-click for full actions.

Replaces replicate-set & bar-group

  • One panel, one mental model. A saved selection is a replicate set; bar plots simply render selections side-by-side in this order.
  • Drag handle (left): reorder. Order here = order in bar plots and legend. ⌥↑/↓ as a keyboard alternative.
  • Eye toggle: hides the selection from all plots without deleting it. Hidden rows fade, strike through the name, and pin to the bottom of the list.
  • Color dot: click to open the curated color row + free-form picker (see §4). Hidden selections always show text-faint until shown again.
  • Inline rename: double-click the name or press F2. Esc cancels, Enter commits.
  • Expand chevron: the row itself expands to show sub-item chips — each well in the selection, with a per-well dot. + add chip lets you append directly without going back to the plate.
  • Kebab / right-click: rename, recolor, duplicate, hide, reorder, export-to-CSV, delete (with confirmation).
  • Footer actions: From selection turns the current plate selection into a new saved row. Import reads a CSV with name,wells,color columns.
Migration. Existing files with separate replicate_sets and bar_groups blobs merge into one selections list on load. Bar-group ordering becomes selection order; replicate-set names become selection names; conflicts append _v2 (and the inline-rename mockup above shows what that looks like in practice).

Color picker & LUT selector

Two related but distinct controls. Review-image colors need a free-form picker (curated palette + custom HSV picker + recents). Review-image LUTs (lookup tables / colormaps) are a long, categorized list that benefits from search and inline preview strips.

Picker · how it composes

  • Curated row first. Six brand-matched swatches cover ~90% of use. Selected swatch gets a 2-px accent outline (offset 2 px).
  • Custom tile. The conic-gradient + "+" tile opens the free-form picker. It's the last item in the row so the curated set stays a quick-pick.
  • Picker structure. SV square (saturation × value) → hue slider → numeric fields (Hex + HSL + Alpha) → recents row. No eyedropper — recents covers that ground without OS-permission complexity.
  • Recents persist per dataset, capped at 8. Click any recent to apply.

LUT selector · how it composes

  • Trigger is a button showing the current LUT's gradient strip + name. Two ghost icons sit beside it: reverse (flips the LUT) and reset (back to default for this channel type).
  • Open state is a popover with a search input at the top and four categories: Perceptual, Diverging, Categorical, Cyclic. Search filters across all categories live.
  • Each row: 60-px gradient strip, monospace name, checkmark for the active LUT. The strip is the real LUT — not a sample image — so what you see is what you'll get on the image.
  • Match count in the header ("3 / 27") gives feedback when the search filter is narrow.
Why split picker from LUT? Picking a color for a trace is single-value. Picking a LUT is choosing a function (intensity → color). Different problems, different UI affordances. Reusing the curated row for LUTs would force tradeoffs that hurt both.

Titlebar — keep custom, finish affordances

Recommendation: keep the custom titlebar. The breadcrumb + file chip + save-status dot are integral to the design language — surrendering them to a native bar fragments the chrome. Below: every previously-missing affordance specified in the v2 vocabulary.

Why keep custom

  • The titlebar carries three load-bearing affordances — breadcrumb, file chip with save-state dot, primary action (Share). None of these can live in a native bar without splitting the chrome.
  • The dark theme is the brand. A platform-styled bar at the top breaks the design's identity on every platform differently.
  • Native bars vary too much across platforms (different fonts, button styles, draggable areas) to design once.

Missing affordances — now specified

  • Min / max / close. Windows/Linux: 28-px icon buttons in a separator-divided group at the far right. Hovered "close" turns danger-red. macOS: traffic lights on the far left in their native colors; min/max are not in the bar (they live on the lights).
  • Resize grips. 4-px transparent strip along the top edge (ns-resize cursor), 6-px on each side (ew-resize), 8-px corners (nwse/nesw). In Qt: invisible child widgets calling windowHandle().startSystemResize(edge).
  • Open dataset. A ghost Open button in the actions area is the primary path. ⌘O is the keyboard path. The brand-logo button (with the small ▾) is the secondary path — it opens an app menu containing Open, Recent files, Preferences, About, Quit.
  • Theme switcher. A sun/moon icon button in the actions area opens the popover shown left. Three tiles (Dark · Light · System) + a "High contrast" toggle. Persists to user settings.
  • Drag area. Anywhere in the titlebar that isn't a button or input. Standard chrome behavior: double-click to maximize, drag to move via startSystemMove().
Fallback. If accessibility audit reveals that the custom bar misbehaves with a screen reader or platform-specific power-user tools (window snapping on Windows, Mission Control gestures on macOS), the v2 titlebar can degrade to native by setting Qt.FramelessWindowHint = False — the breadcrumb and actions move to a second 36-px strip below the OS bar. Mockup for that fallback exists in the PyQt6 notes appendix.
